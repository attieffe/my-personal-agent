## Multi-utente su OpenClaw: le opzioni concrete

OpenClaw **non ha un sistema ACL per-utente nativo**, ma offre diversi agganci che si possono combinare. Ecco le opzioni dalla più semplice alla più robusta:

---

### Livello 1 — Prompt-based ACL (zero codice, immediato)

Crei un file `PERMISSIONS.md` nel workspace con una tabella:

```
| sender_id    | nome    | può fare                        | non può fare          |
|--------------|---------|---------------------------------|-----------------------|
| 506258994    | Atti    | tutto                           | -                     |
| 111222333    | Chiara  | myJob/PERSONALE, padel          | email, finanza, files |
| 444555666    | Collega | solo rispondere domande         | tutto il resto        |
```

E nel `AGENTS.md` (o `SOUL.md`) aggiungi:

> "All'inizio di ogni turno, leggi sender_id dall'inbound metadata. Confrontalo con PERMISSIONS.md. Se l'utente non ha accesso a qualcosa che sta chiedendo, rifiuta educatamente."

**Pro:** immediato, zero codice, flessibile  
**Contro:** sicurezza "best effort" — un utente malizioso potrebbe fare prompt injection. Per **utenti fidati** (familiari, colleghi) è più che sufficiente.

---

### Livello 2 — Config OpenClaw: routing per chat_id (supportato nativamente)

OpenClaw supporta già due cose utili:

**a) Modello diverso per chat_id:**
```json5
channels: {
  modelByChannel: {
    telegram: {
      "506258994": "anthropic/claude-sonnet-4-6",  // Atti → Sonnet
      "111222333": "openai/gpt-5.4-mini",          // altri → modello più economico
    }
  }
}
```

**b) Bindings ACP per chat/peer** — puoi bindare peer specifici a sessioni con workspace diversi (richiede acpx plugin). Ogni "utente" avrebbe il suo AGENTS.md isolato.

**Limite:** non isola i tool completamente, il workspace fisico è lo stesso.

---

### Livello 3 — Istanze multiple OpenClaw (isolamento totale)

Un bot Telegram diverso per ogni "ruolo":
- **Bot principale** (solo tu) → workspace attuale, tutti i tool
- **Bot famiglia/colleghi** → workspace dedicato, AGENTS.md limitato, tool policy ristretta

Ogni istanza è un processo OpenClaw separato con config propria. Isolamento totale, nessun rischio di crossover.

**Pro:** sicuro, pulito, ogni bot ha il suo carattere/nome  
**Contro:** gestire più processi, più token Telegram

---

### Livello 4 — Plugin/middleware custom (sviluppo Node.js)

Un plugin OpenClaw che intercetta ogni messaggio prima che arrivi all'agente, verifica il sender_id contro una ACL, e blocca/riduce i tool disponibili a runtime.

Questo è l'unico modo per avere **enforcement reale** senza istanze multiple. Richiede:
- Conoscenza dell'API plugin di OpenClaw
- Sviluppo Node.js (~200-300 righe)

## Il nodo critico: come passa il sender_id in modo trusted?

C'è un dettaglio importante. I parametri dei tool MCP li decide l'agente — quindi se passi `user_id` come parametro del tool, un utente furbo potrebbe far passare all'agente un `user_id` sbagliato.

**Soluzione:** il `sender_id` non deve venire dai parametri del tool, ma dagli **headers HTTP** della chiamata MCP. OpenClaw supporta headers statici per server MCP:

```json5
mcp: {
  servers: {
    expense-tracker: {
      url: "https://tuoserver.com/mcp",
      headers: { "X-User-Id": "506258994" }
    }
  }
}
```

Il problema: gli headers nella config sono statici — tutti gli utenti userebbero lo stesso. Quindi hai due strade:

**Strada A — Token per utente nella config** (richiede una config/istanza OpenClaw per utente): ogni utente ha `Authorization: Bearer <token-personale>` → il server MCP identifica l'utente dal token. Sicurezza reale.

**Strada B — user_id come parametro con trust parziale** (singola istanza): l'agente passa il sender_id dall'inbound metadata, il server MCP lo usa ma non può verificarne la provenienza. Funziona per team fidato, non per ambiente ostile.
