**Branch per ogni task (consigliato)**

Un task = una branch = una PR. Tu fai il merge quando sei pronto, nel tuo tempo. Vantaggi: ogni intervento è isolato, puoi approvarne uno e bloccare un altro, la storia git è pulita. Svantaggio: se hai 5 task aperti insieme, ti ritrovi 5 PR da gestire.

**Branch `dev` fissa**

L'agente accumula tutto lì, tu mergi `dev → main` quando hai controllato tutto insieme. Più semplice da gestire mentalmente, ma perde l'isolamento — se una modifica è sbagliata, non puoi selezionare cosa mergiare e cosa no.

**Il mio consiglio**

Per il tuo caso — repo che conosci, interventi mirati — la branch per task è meglio. Ma con una regola operativa: **una cosa alla volta**. Tu dai un task, l'agente lavora, apre la PR, ti notifica. Tu approvi o chiedi fix. Solo dopo parte il task successivo. Eviti il problema delle 5 PR aperte e mantieni il controllo senza caos.

Il merge lo fai tu. Non perché l'agente non possa tecnicamente farlo, ma perché quel momento è il tuo checkpoint reale — è lì che decidi "questo entra in produzione".
